= Git Repository Management

Git repository management encompasses the practices, configurations, and workflows needed to effectively maintain version-controlled projects. This document covers common configuration concerns and troubleshooting topics.

[IMPORTANT]
====
Ensure you are using the correct Git package (`Git.Git`) before applying these configurations. Avoid standalone `Git.GCM` or `Microsoft.Git` packages. Refer to xref:reference/git-installation.adoc[Git Installation] for details.
====

For complex topics that warrant deeper coverage, see the related documents linked below.

== Line Endings Configuration

Git warns about line ending inconsistencies when files in your working directory use different line endings than what Git expects. Windows typically uses **CRLF** (Carriage Return, Line Feed) line endings, while Linux and macOS use **LF** (Line Feed). If Git's line-ending behavior doesn't match your editor and repo (e.g. `core.eol` unset on Windows), you can get **false diffs** — every line showing as changed due only to CRLF vs LF, with no real content change.

Git's `core.autocrlf` setting controls automatic conversion:

* `true` (Windows default): Converts LF to CRLF on checkout, CRLF to LF on commit
* `input`: Converts CRLF to LF on commit only (no conversion on checkout)
* `false`: No conversion; use with `core.eol` to define what Git expects in the working tree

The `core.eol` setting (values: `lf`, `crlf`, or `native`) tells Git which line ending to use in the working tree when `core.autocrlf` is `false`. Setting it explicitly avoids ambiguity and false diffs.

=== When and Why Use `core.autocrlf true`

Use `core.autocrlf true` only when you need **CRLF in your working directory** for tooling that expects it (e.g. some legacy Windows-only scripts or editors). Git then converts to LF on commit so the repository stays LF. Downsides: conversion on every checkout/commit can introduce false diffs or merge noise if other tools (editors, formatters) also touch line endings, and it couples your workflow to Git's conversion instead of keeping one canonical ending everywhere.

=== Recommended: `autocrlf false` and `core.eol lf`

For most development (especially cross-platform repos and modern editors), prefer **no conversion** and a single canonical ending: **LF**. That keeps the working tree and the repository in sync and avoids false diffs from CRLF/LF mismatches.

==== Step 1: Set `core.autocrlf` to `false` and `core.eol` to `lf`

.Global setting (affects all repositories):
[source,bash]
----
git config --global core.autocrlf false
git config --global core.eol lf
----

.Repository-specific setting:
[source,bash]
----
git config core.autocrlf false
git config core.eol lf
----

==== Step 2: Normalize the Repository (if changing from a previous setting)

After changing the setting, normalize the repository so the index and working tree use LF:

[source,bash]
----
git rm --cached -r .
git reset --hard HEAD
----

This removes files from the staging area (`git rm --cached -r .`) and then restores the working directory to the last committed state (`git reset --hard HEAD`), applying the new line-ending behavior on checkout.

==== Step 3: Use LF in Your Editor

Configure your editor to use LF so new and saved files stay LF and match Git. In VS Code: set **`Files: Eol`** to **`\n`** (File > Preferences > Settings, search for "files: eol"). Other editors have similar options.

[NOTE]
====
Editor Eol only affects new files or files when you save. For existing files, change line endings in the editor (e.g. in VS Code use the status bar "CRLF"/"LF" control) if needed.
====

==== Step 4: Use EditorConfig for Cross-Editor Consistency

Put a `.editorconfig` in the project root to force EOL across all editors (VS Code, Visual Studio, JetBrains, etc.):

[source,ini]
----
[*]
end_of_line = lf
insert_final_newline = true
----

Some Windows-specific files must use CRLF. Add overrides for them:

[source,ini]
----
[*.bat]
end_of_line = crlf

[*.cmd]
end_of_line = crlf

[*.reg]
end_of_line = crlf
----

See xref:reference/windows-eol-exceptions.adoc[Windows EOL Exceptions] for details.

=== Alternative: `core.autocrlf input`

If you want Git to normalize line endings only on commit (CRLF→LF) and not on checkout, set `core.autocrlf` to `input`. Your working tree can then stay CRLF on Windows if you prefer; the repository will store LF.

**When to use `input` instead of `false`:** On Windows, many tools (VS Code, Visual Studio, JetBrains, Git for Windows installer) default to CRLF. If you prefer to avoid forcing LF in your working tree and only want commits normalized to LF, `input` is a good choice. It normalizes on commit while leaving checkout behavior alone.

.Global:
[source,bash]
----
git config --global core.autocrlf input
----

After changing, run the same normalize steps (Step 2 above) if the repo had mixed endings.

=== Suppress the Warning (Not Recommended)

If you understand the issue and accept the automatic conversion, you can suppress the warning by setting `core.safecrlf` to `false`. This masks inconsistent commits and is not recommended.

[source,bash]
----
git config --global core.safecrlf false
----

=== Line Endings Summary

[cols="1,2,2"]
|===
| Setting | Value | Effect

| `core.autocrlf`
| `true`
| Convert LF→CRLF on checkout, CRLF→LF on commit (Windows default). Use only when tooling requires CRLF on disk.

| `core.autocrlf`
| `input`
| Convert CRLF→LF on commit only; no conversion on checkout.

| `core.autocrlf`
| `false`
| No conversion. Pair with `core.eol lf` (recommended) to avoid false diffs and keep working tree LF.

| `core.eol`
| `lf`
| Working tree uses LF; use with `core.autocrlf false` for a single canonical ending.

| `core.safecrlf`
| `false`
| Suppress line ending warnings (not recommended)
|===

== Default Branch Configuration

Git has announced plans to change the default initial branch name from `master` to `main` in a future release. The `init.defaultBranch` configuration variable allows you to specify your preferred default branch name for new repositories.

=== Setting the Default Branch Name

To ensure consistency with modern conventions used by GitHub, GitLab, and other platforms, set your default branch to `main`:

[source,bash]
----
git config --global init.defaultBranch main
----

=== Background and Timeline

* **Introduced in:** Git 2.28.0 (July 2020)
* **Current status:** The default remains `master`, but Git's documentation marks it as "subject to change in the future"
* **Industry adoption:** GitHub, GitLab, and most major Git hosting platforms have already transitioned to using `main` as the default

=== Why Configure This Now?

1. **Future-proofing:** When Git eventually changes the default, your workflow will remain consistent
2. **Collaboration consistency:** Most new repositories on GitHub and GitLab now use `main` — aligning your local defaults prevents confusion
3. **Documentation alignment:** Modern tutorials and documentation increasingly assume `main` as the default branch

[TIP]
====
This setting only affects *new* repositories created with `git init` after the configuration is applied. Existing repositories retain their current branch names.
====

== CLI Editor & Merge Tool Configuration

On modern Windows 11 systems, we recommend using Microsoft's `edit` command-line editor as the default Git editor and merge editor in the terminal. `edit` is lightweight, ships with a familiar UI, and avoids the complexity of modal editors.

.Install `edit` on Windows (if needed)
[source,powershell]
----
winget install Microsoft.Edit
----

.Set `edit` or VSCode (`code`) as the default Git editor
[source,bash]
----
git config --global core.editor edit
# or
git config --global core.editor code
----

.Set `meld` as the default merge editor (or find one you like first - you can even use vscode)
[source,powershell]
----
# Install Meld
winget install Meld.Meld
----

[source,bash]
----
git config --global merge.tool meld
git config --global mergetool.edit.cmd 'meld "$MERGED"'
git config --global mergetool.prompt false
----

[NOTE]
====
`edit` runs directly in your terminal session, so Git naturally waits for it to close before continuing (no special `--wait` flag is required). The simple mergetool configuration above opens the merged file with conflict markers; you resolve conflicts manually in `edit`, save, and then let `git mergetool` complete.
====

== Remote Naming Conventions

By default, Git uses the name `origin` for the remote from which a repository was cloned. This terminology is relative to the local machine (essentially meaning "the remote place I push to") rather than reflecting the actual structure of the project ecosystem. To improve clarity and standardize terminology across our workflow, we recommend using absolute terminology that reflects the fork structure.

=== Standard Remote Names

When working with forks, configure your remotes as follows:

* `origin`: The absolute source of truth for the project (often traditionally called `upstream`). This is the main repository owned by the organization or project maintainers. Reassigning `origin` to mean the absolute origin of the codebase reduces cognitive load.
* `personal` (or your username): Your personal fork of the repository. You push your feature branches here to validate changes (e.g., triggering personal Vercel deployments or running CI/CD pipelines) before opening a pull request to `origin`.

=== Example Configuration

When you clone your personal fork, Git automatically names it `origin`. You should immediately rename it and add the true `origin`:

[source,bash]
----
# 1. Rename your personal fork from 'origin' to 'personal'
git remote rename origin personal

# 2. Add the main project repository as 'origin'
git remote add origin git@github.com:Organization/project-name.git

# 3. Verify your remotes
git remote -v
----

Expected output:
[source,text]
----
origin    git@github.com:Organization/project-name.git (fetch)
origin    git@github.com:Organization/project-name.git (push)
personal  git@github.com:YourUsername/project-name.git (fetch)
personal  git@github.com:YourUsername/project-name.git (push)
----
