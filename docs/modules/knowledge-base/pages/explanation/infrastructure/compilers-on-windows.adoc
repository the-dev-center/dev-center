= Compilers on Windows: Clang vs. GCC
:navtitle: Compilers on Windows

When developing high-performance software on Windows, selecting the right compiler toolchain is critical for both runtime performance and developer experience.

== Why Clang is Preferred

For modern Windows development, **Clang** (part of the LLVM project) is generally preferred over GCC for several reasons:

* **Native Compatibility**: Clang is designed to be compatible with Microsoft's Visual C\+\+ (MSVC) ABI. This means binaries compiled with Clang can interoperate seamlessly with native Windows libraries and tools.
* **Superior Diagnostics**: Clang providing significantly clearer and more actionable error messages and warnings compared to traditional GCC versions.
* **High Performance**: Clang is built for high-performance software, often providing better optimization passes for modern CPU architectures.
* **Tooling Integration**: LLVM-based tools (like `clang-format`, `clang-tidy`, and LLD) provide a cohesive ecosystem that integrates well with modern IDEs and CI/CD pipelines.

== The GCC & MSYS2 Confusion

While GCC is the standard on Linux, using it on Windows often leads to a convoluted environment:

* **The MSYS2 Necessity**: To use GCC to compile native Windows executables (via MinGW-w64), you typically need **MSYS2**.
* **Complexity & Confusion**: MSYS2 provides multiple "environments" (UCRT64, MINGW64, CLANG64, MSYS). This often confuses developers who find themselves accidentally linking against the POSIX-compatibility layer (`msys-2.0.dll`) instead of creating truly native Windows binaries.
* **Path Pollution**: MSYS2 environments often rely on complex shell configurations and path management that can conflict with other Windows-native tools.

== Linker Considerations

The compiler is only half the story; the linker determines how your binary interacts with the system:

* **MSVC Linker (`link.exe`)**: When using `clang-cl` (Clang's MSVC-compatible driver), you typically use the Microsoft linker. This provides the best compatibility with Windows SDKs and proprietary libraries but requires the **Visual Studio Build Tools**.
* **LLVM Linker (`lld-link`)**: LLVM's own linker is much faster than `link.exe` and is highly compatible with the MSVC ABI. It is included in the `LLVM.LLVM` package as `lld`.
* **The MinGW `ld` Trap**: Some build systems (like those ported from Linux) may still insist on using the `ld` linker from MinGW. If you find yourself needing `ld`, you are often forced back into the MSYS2 ecosystem, which reintroduces the compatibility and complexity issues mentioned above. We recommend using `lld` as a modern, high-performance alternative whenever possible.

== Why Clang may still ask for `link` or `ld`

After installing `LLVM.LLVM` via winget, you might be surprised to see errors like `link.exe not found`. This happens due to a combination of **defaults, drivers, and missing dependencies**:

*   **Default "System" Linker Search**: Clang is designed to be a "drop-in" replacement for a platform's standard compiler. On Windows, it tries to act like MSVC and looks for `link.exe`. In a MinGW environment, it acts like GCC and looks for `ld.exe`.
*   **The "Secret" Dependency (System Libraries)**: While `LLVM.LLVM` provides the compiler and linker, it **does not** include the Windows System Headers or the C Runtime Libraries (e.g., `libcmt.lib`). To get these, you still need the **Windows SDK** or **Visual Studio Build Tools**.
*   **Driver Choice (`clang-cl`)**: LLVM includes `clang-cl.exe`, which mimics Microsoft's `cl.exe`. While it is better at finding the Windows SDK automatically, it is hard-coded to prefer `link.exe` or `lld-link.exe`.

=== Forcing LLVM's Linker

To bypass the search for external linkers and use the high-performance `lld` included with your LLVM installation, use the `-fuse-ld` flag:

[source,powershell]
----
# Force Clang to use its internal lld linker
clang -fuse-ld=lld main.c
----

== Recommended Toolchain

We recommend using the official LLVM distribution for Windows. It provides a clean, native experience without the overhead of a POSIX-emulation environment.

[source,powershell]
----
# Install the full LLVM toolchain via winget
winget install LLVM.LLVM
----

This installation includes `clang`, `clang++`, and the `lld` linker, which are ready to use from any standard PowerShell prompt.
